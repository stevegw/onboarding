01 — The Starting Point

Good documentation.
Hard to learn from.

As someone who has worked in enablement for a long time, I've seen how users actually relate to technical documentation. With Windchill — and PTC products generally — the pattern is familiar: the content is thorough and accurate, but it is structured for people who already understand the system, not for people still building that understanding.

Topics are described in isolation. The web help rarely explains how different parts of the system connect, or what a user is actually trying to achieve when they land on a page. The result is documentation that answers precise questions well, but doesn't help someone develop working knowledge of how everything fits together.

This is the gap I wanted to explore using AI — not to replace the documentation, but to see whether it could help bridge the distance between reference material and real understanding, across multiple PTC products and potentially across languages.

A note on the approach: I ran all of these experiments myself. Being able to evaluate each result against experience of what good enablement looks and feels like — what flows, what confuses, what learners actually need — was central to assessing what was and wasn't working at each stage.

02 — The Web Scraping Experiments

An idea that seemed right — and what it actually taught me

My first instinct was to use the web help pages as raw material. Scraping them programmatically felt like an elegant solution — the content was already there, AI could restructure it into something more learnable. I tried this three times, each version a direct response to what the previous one revealed.

Version 1
Basic scraping + AI restructuring

What I tried

Scraped Windchill web help pages and passed the content through the Claude API to generate scenario-based challenge cards — each with a question, context, progressive hints, a worked answer, and text narration to support different learning preferences.

What I found

  • The pipeline worked technically and produced content quickly.
  • Output quality was inconsistent. Questions were sometimes off-target; answers occasionally missed the mark.
  • The web help's reference-style writing didn't give AI enough context to generate reliably useful learning material.
Version 2
Refined prompting + local LLM

What I tried

Introduced a local LLM (Ollama/Mistral) to explore whether on-device AI could improve adaptability and reduce external dependency. This was a deliberate test of the concept of dynamically generated learning content — could AI respond contextually and adapt in real time?

What I found

  • The concept of dynamic AI-generated learning was validated — it is technically feasible and worth exploring further.
  • Smaller local models produced noticeably lower quality output than Claude for this type of content.
  • The approach became too abstract to pursue further at this stage, but it proved the idea has legs.
Version 3
Further scraping iteration

What I tried

A third attempt at refining the pipeline — better pre-processing of scraped content before passing to AI, trying to address the consistency issues that had persisted across the first two versions.

What I found

  • Incremental improvement, but the fundamental problem remained.
  • The source material was the ceiling, not the pipeline or the prompting.
  • No refinement could fully compensate for web help content that lacked the depth and context needed for reliable learning output.

Three attempts at web scraping produced a clear conclusion: the quality of the output is determined almost entirely by the quality of the input. This wasn't a failure — it was a necessary experiment that pointed clearly to the next step.

03 — The Content Quality Pivot

When the source improved,
everything improved.

Moving to official PTC PDF documentation as the source material changed the results noticeably. PDFs contain more complete explanations and proper context — they were written with some thought for the reader. That gave AI genuinely useful material to work with and the learning output reflected that difference immediately.

The focus also shifted from generating quizzes toward building interactive learning guides — content structured around how a user thinks through a task, with a clear flow from start to finish. That approach consistently landed better than anything produced from the scraped web content.

📋
PDF as the source of truth
Official Windchill, Codebeamer and Creo PDFs gave AI the context needed to produce accurate, complete learning content rather than reformatted reference text.
🔁
Learning flow over feature lists
Guides built around what a user is actually trying to do — with a beginning, middle and end — were consistently more usable than content mirroring documentation structure.
🔓
Accessible by design, no IT dependency
Platform decisions throughout were deliberately low-friction — browser-based, no login, no LMS infrastructure. The goal was to validate ideas quickly, without organisational overhead.

04 — What the Current Solution Looks Like

A working system — not just a prototype

The current solution isn't a finished product, but it's considerably more than an experiment. It has a repeatable build process, an authoring layer for ongoing maintenance, and the ability to reach a global audience. What's here now is enough to understand what a fuller version would look like.

⚡
~25 minutes to build a new English module
Once the source PDF is selected and the structure is defined, generating a complete learning module for a new topic takes around 25 minutes. That's a meaningful shift in what's feasible for a small enablement function without dedicated content production resource.
Not all modules have been built yet — the effort has been on proving the process rather than achieving full product coverage.
✏️
Authoring tools built in
The current solution includes an authoring feature set that allows adding images, updating body text, and making content corrections without regenerating from scratch. This means the content can evolve as products change or as feedback comes in, without starting over each time.
Keeping a human in the loop for content maintenance was a deliberate choice — AI generates, but the enabler validates and refines.
🌍
AI translation to five additional languages
Content produced in English can be translated by AI into French, German, Spanish, Chinese and Japanese. The same source material, authored once, can reach a global team without separate writing effort for each language — though translation quality would benefit from native-speaker review before formal deployment.
🧩
Built for four PTC product areas
Modules have been developed across Windchill Admin, Windchill BOM, Codebeamer and Creo Fundamentals. This demonstrates that the approach isn't specific to one product — the same process applies wherever there is good PDF source material to work from.
The breadth was intentional — testing generalisability mattered as much as depth within any single product.
~25min
That's the time to take a new topic from PDF source to a complete interactive English learning module. Translate that into five additional languages with one additional step. For a lean enablement function, that changes what's achievable without additional headcount or tools budget.
05 — What I've Concluded

AI works best when it works
alongside human expertise —
not instead of it.

Finding 01
Source quality is everything
The scraping experiments made this undeniable. AI amplifies what's in the source material. Shallow, context-free input produces shallow output. Verified, well-written PDF documentation produces genuinely useful learning content.
Finding 02
Dynamic AI-generated learning is a real possibility
The local LLM experiment confirmed the concept even if that specific path wasn't the right one to continue. Contextual, responsive AI-generated content is feasible — the question is how to do it reliably and at the right quality level.
Finding 03
The enabler's role doesn't disappear — it changes
Experienced judgement was needed at every stage: recognising when output was off, knowing what good learning design looks like, deciding what to fix and what to keep. AI accelerates the work; it doesn't replace the expertise that guides it.
Finding 04
Scale and reach are suddenly within reach
A 25-minute build time per module, AI translation into five languages, and an authoring layer for maintenance — these aren't theoretical. They're what the current solution can already do. That's a different conversation about enablement capacity than was possible before.
The central conclusion

The most interesting possibility isn't AI replacing content creation — it's AI and human content expertise working in harmony. The human brings validated knowledge, learning design judgement, and quality control. AI brings scale, translation, accessibility, and the ability to make that knowledge available at the moment it's needed.

On the broader question

This exploration was genuinely about understanding where AI fits — not proving that it does. What it's shown me is that the value isn't in automating enablement away. It's in using AI to extend what a skilled enabler can do: turning good source material into learning experiences that reach more people, more often, in more languages, with far less effort per module. What the organisation chooses to do with that understanding is a separate conversation.

06 — Questions Worth Asking Next

The exploration surfaced
some useful open questions

These aren't recommendations — they're honest questions that follow from what the experiments revealed. Where the organisation wants to take them is a separate conversation.

💬
Where do users go today when they're stuck? Understanding this current behaviour — across products and geographies — is the most important thing before deciding how or whether to change it.
🌍
How important is multilingual reach for our user base? AI translation is already working across five languages. Understanding how much of the audience would benefit helps prioritise whether this is worth investing in properly — including native-speaker review.
🤝
How should content creators and AI tools work together in practice? The experimentation suggests a productive model — authored by humans, scaled by AI, validated before release. But the workflow needs to be designed deliberately.
📋
Which products and topics should be prioritised? The build process is established and takes around 25 minutes per English module. The question now is where to direct that effort first — and who decides.
🔄
How does the content stay current? The authoring tools allow updates without regenerating from scratch. But ownership of keeping content accurate as products evolve still needs to be defined.
